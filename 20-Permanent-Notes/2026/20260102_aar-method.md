# Key Points

- 米軍で開発されたPDCAサイクルのための振り返りフレームワーク
- 次にやることを明確にするために使う

# Context
- 定期的な振り返りを行うことにより、自分の現在地を知ることができる
- 過程を追うことでその過程が正しいものだったかを検証することができる
- 検証結果をもとに、次に取るべきアクションを知ることができる

# Body
AARでは以下の4つの質問で構成される整理を行う。

- 何を達成しようとしたか（計画）
	- 実際に達成しようとした目標を記述
	- プロジェクト固有のものでも自分のキャリアに対する抽象的なものでも
- 実際に何が起こったか（実績）
	- 結果や実績を記述する
	- できなかったとしても記述
- なぜそうなったか（原因分析）
	- できた要因、できなかった原因を分析
- 次回どうするか（改善策）
	- SMARTな目標を立てる 


## 実施のポイント

### 1. 何を達成しようとしたか（計画）
- **目標の明確化**: 当初設定した目標を具体的に記述
- **背景の確認**: なぜその目標を設定したのかという文脈も含める
- **期待値の設定**: どのような成果を期待していたかを明記

**例**:
- プロジェクト：「新規顧客獲得キャンペーンで、3ヶ月間で50社の新規契約を獲得する」
- キャリア：「データ分析スキルを向上させ、より戦略的な提案ができるようになる」

### 2. 実際に何が起こったか（実績）
- **客観的な記述**: 感情や解釈を交えず、事実のみを記録
- **定量データ**: 可能な限り数値で表現
- **成功と失敗の両方**: ポジティブな結果もネガティブな結果も記載

**例**:
- プロジェクト：「3ヶ月間で新規契約32社を獲得。目標の64％達成。第1ヶ月15社、第2ヶ月10社、第3ヶ月7社と減少傾向」
- キャリア：「Pythonでのデータ分析研修を修了。実務で3件の分析レポートを作成したが、経営層への提案には至らず」

### 3. なぜそうなったか（原因分析）
- **成功要因の特定**: うまくいった理由を分析
- **阻害要因の把握**: 目標未達の原因を特定
- **外部環境の影響**: 自分でコントロールできない要因も考慮
- **自己の行動分析**: 自分の判断や行動がどう影響したか

**例**:
- 成功要因：「第1ヶ月は既存顧客からの紹介が多く、信頼関係が活きた」
- 阻害要因：「第2ヶ月以降、フォローアップの仕組みが不十分で、見込み客の離脱が多発。また、競合他社の新サービス投入により価格競争が激化」
- 自己分析：「データ分析は習得したが、結果を経営視点でストーリー化するスキルが不足していた」

### 4. 次回どうするか（改善策）
- **具体的なアクション**: 抽象的な反省ではなく、実行可能な行動を定義
- **SMARTな目標**: 次の行動は測定可能で期限のある目標として設定
- **優先順位**: 複数の改善策がある場合、優先度をつける
- **学びの適用**: 今回の経験から得た知見を次にどう活かすか

**例**:
- 「2026年4月から、見込み客に対して週1回の定期フォローメールを自動配信する仕組みを導入する」
- 「2026年3月末までに、プレゼンテーション研修を受講し、経営層向けの提案資料作成スキルを習得する」
- 「次の四半期は、既存顧客満足度を重視し、リピート率を現在の45％から60％に向上させる」

## AARの実施タイミング

- **プロジェクト完了時**: 大きな節目での振り返り
- **定期的な実施**: 週次、月次、四半期ごとの習慣化
- **重要な意思決定後**: 判断の適切性を検証
- **予期しない出来事の後**: 失敗や想定外の成功からの学び

## Related Concepts

- PDCAサイクル - 継続的改善フレームワーク
    - AARはPDCAのCheck（評価）とAct（改善）を構造化する手法として機能
- SMARTフレームワーク - 目標設定手法
    - AARの「次回どうするか」で具体的な目標を設定する際に活用
- KPT法（Keep/Problem/Try） - 振り返り手法
    - より簡易的な振り返り手法。AARはより詳細な原因分析を含む
- 内省（リフレクション） - 学習手法
    - 経験から学ぶための基盤となる思考プロセス

# 応用例

## ビジネスプロジェクトでの応用
**新製品ローンチプロジェクト**
- 達成目標：「2025年Q4に新製品をリリースし、初月100万円の売上を達成」
- 実績：「予定より2週間遅れでリリース。初月売上75万円」
- 原因：「開発フェーズで仕様変更が多発。マーケティング開始が遅れた」
- 改善策：「次回は仕様凍結期限を設け、並行してマーケティング準備を開始」

## 個人の学習・キャリアでの応用
**資格試験への挑戦**
- 達成目標：「6ヶ月間の学習で簿記2級に合格する」
- 実績：「試験は不合格（得点率68％、合格ライン70％）」
- 原因：「前半3ヶ月の学習時間が不足。後半は詰め込みで理解が浅かった。工業簿記の演習量が特に不足」
- 改善策：「次回は毎週10時間の学習時間を確保。特に工業簿記の問題演習を週5問以上実施。3ヶ月後の試験で合格を目指す」

## チーム運営での応用
**スプリント振り返り（アジャイル開発）**
- 達成目標：「2週間スプリントで予定した15のストーリーポイントを完了」
- 実績：「実際には12ポイントを完了。3ポイント分は次スプリントに繰越」
- 原因：「技術的負債の対応に予想以上の時間を要した。また、依存する外部APIの仕様変更が発生」
- 改善策：「技術的負債対応用に毎スプリント20％のバッファを確保。外部依存の確認を計画段階で徹底」
